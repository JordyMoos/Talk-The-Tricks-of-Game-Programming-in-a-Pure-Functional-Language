Game Loop

Which we call the Game Loop
A very simplistic one is this, update the game, and then sleep for some time
And keep doing that forever
Code samples with white background are pseudo code in a C like style

This update function will be called once a second.
Which you can use to update your game state.
Now if we let the actors move one tile each update then that will look like this
I added walls so our update logic can use that to reverse direction
The game moves very slow and we can fix that to lower the sleep time.
17 milliseconds will give us about 60 updates per second
Now we are not really calling update every 17ms.
We forgot to take into account the time it took to run `update`
If update takes about 10ms then together with sleep we only call update about once in the 27ms
You can fix that by subtracting the time it took from the sleep
Another option would be to check how much time has passed since the last update
And use that as some kind of delta time in update
That within update you check how much actors must move based on the delta time
An disadvantage of this is that too much time might have passed. And actor might jump pass walls
A safer bet is to have a stable update call. Which makes it unable for your game to make giant loops on slow computers
If you split the update and view. You can do multiple update ticks in case too much time has passed the last loop.
To let the game `catch up`. At the cost of drawing the game less frequent. Which we will see as "frame drops"
Elm is a pure functional language and asking for the time is an impure operation.
Time has this habit of changing a lot
In order to get some time information you need to setup something called "Subscriptions"
Which allows mechanisms outside of your app to send `Messages` which you can handle in a pure fashion.
We define a message, setup a subscription to get that message every 17 milliseconds
And handle the message
We are running in the browser and the browser has some mechanism to redraw the screen.
And to make animations look smoother we can kinda hook into that redraw mechanism
We can do something similar as we did with time but then based on the browsers animation frame
This time we get the amount of milliseconds passed instead of a "stable 17ms" like we had with time
And therefor we should be doing some match instead of calling the update function everytime
Remembering our game loop. We would like to update on a stable 17 millisecond. For as many 17 milliseconds since we did our last loop
You do not have for loops in Elm but we can create a List of a certain length and iterate over it
Which we do with List.repeat
We determine how often our game speed has passed and create a list of the same length
Because we need to update the game based on the previous update. We can fold over our list and use our model as the accumulator
This will give us the sketched game loop
We can improve the game loop by limiting the amount of updates per "view".
We do this to not make it look like our hero jumped far because of a game hickup
This is easy by just taking x elements from our repeated list
Finally we should keep track of how many time we had left. Like 3ms if the timeSinceLast was 20ms.
After the `update` function returned our new model, elm will call our view function again
In which we will render the game

Actors

A way to have actors is to create the individuals.
Like we have a Hero and an Enemy
Which both have an `update` function.
Which in turn gets called from the main loop
You add all the behaviours in those classes like walking and collection diamonds for our hero.
And walking and killing heroes for the enemy
After a while you will end up with more and more actors
Like a butterfly, which is an AI just like our Enemy but then being super cute instead of trying to kill the hero
This goes on and on and you end up with lots of actors which update functions get really big and messy
Also you will notice you get lots of duplicate behaviour among your actors.
A solution in the object oriented world could be to introduce some form of inheritance
Which is actually not really reusable and might give us "Diamonds of deaths"
Like I have no clue how this update function would look like
Another more reusable approach would be to put the behavior in separate places, which we called components here
And assign components to actors. Like our hero has the WalkComponent and CollectComponent or whatever
And we would give our enemy the KillComponent.
This way we can easily mix and match components and thereby create new actors with less effort

Composition over Inheritance

The benefits of Composition over Inheritance
This is nice because we do not have inheritance in Elm anyway hehe
Elm can do the composition which we call the Actor Component system
In Elm we can create our own type which we call Component
And Component can by any of those specified options
Its a bit like an Enum.
Components can hold data like we give a String to TagComponent to represent the tag name
And its can also have custom type or type aliases. Like we create some TransformComponentData record with x and y positions for the transform.
Once we have our components we can mix and match them to create actors.
We define an actor as just a list of components and we create our hero and enemy.
Now a little sidetrack
ControllerInputComponent and WalkAiComponent kinda do the same thing. They control the actors movement etc
Those components just do it in their own way, via collected input or some ai logic
We could group them together under one ControlComponent to make it clearer that you should not have both

Now the fun stuff begins

Which will just make the actor move to the right
We receive the actor that holds this component and we will net to fetch the transformComponent from the actor
Because the transform holds its current position
After that we will walk one position to the right. Which will create a new transform component
Which in turn needs to overwrite the current transform component of the actor
Finally if the actor did not have a transform component then we will return the actor as we received it
This function will try to get the TransformComponentData from the actor. But he might not have any therefor we return a Maybe.

walkRight will just create a new TransformData based on the current one but with x increased by one
